# Wiki 프로젝트 규칙

## Git 워크플로우

- **main 브랜치에 직접 커밋/푸시 금지**
- 모든 변경사항은 feature 브랜치를 만들어서 작업
- 작업 완료 후 PR 생성 → 리뷰 → 머지 순서로 진행

### 브랜치 네이밍
```
docs/주제-내용
예) docs/network-http-basics
    docs/database-index
    docs/backend-rest-api
```

### PR 프로세스
1. feature 브랜치 생성
2. 문서 작성/수정
3. PR 생성 (제목: 어떤 내용을 추가/수정했는지 명확하게)
4. 리뷰 후 머지

## 문서 작성 원칙

학습 자료나 문서를 기반으로 내용을 정리할 때, 단순히 문서 내용을 나열하지 않는다.

- **실제 사용 사례 포함**: 해당 기술이 실무에서 어떻게 쓰이는지 구체적인 상황과 함께 설명
- **코드 예시 필수**: 개념 설명 후 반드시 실제 동작하는 코드 예시 첨부
- **Good / Bad 패턴**: 가능하면 잘못된 사용법과 올바른 사용법을 비교해서 설명
- **왜(Why) 중심**: 무엇인지(What)보다 왜 이렇게 하는지(Why)를 중심으로 설명
- **튜토리얼 마무리**: 각 문서 마지막에는 배운 내용을 직접 적용해볼 수 있는 실습 과제나 미니 프로젝트 아이디어를 제시
